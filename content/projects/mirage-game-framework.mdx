## Overview

A C++ gameplay framework for Unreal Engine 5.7, shipped as one multi-module
plugin. It covers the systems most action RPGs need: stats and damage,
inventory, equipment, combat, hit detection, loot, interaction, AI, audio, and
camera. Each is built once, networked correctly, and made to extend rather
than rewrite.

It consolidates two earlier plugins (inventory and combat) into one
architecture, and it's the base I build other games on.

## What it handles for you

- **Replication done right.** Every inventory, equipment, container, stat, and
  interaction list uses a `FFastArraySerializer`, so only deltas replicate and
  client UI updates are event-driven instead of polled. A lag-compensation
  subsystem rewinds hitboxes so shots land where the client saw them.
- **GAS wired up.** Equipment can apply Gameplay Effects, grant abilities, and
  drive animation layers through the Ability System Component, for players and
  AI alike. A stat↔GAS bridge keeps replicated stats and attributes in sync.
- **AI out of the box.** StateTree behaviors with perception, suspicion,
  awareness, squads, patrols, and hit reactions — coordinated by an AI
  director that paces encounters, with significance and dormancy keeping large
  populations cheap.
- **No brittle coupling.** Systems talk through gameplay tags, delegates, a
  gameplay message subsystem, and runtime component resolution, so one module
  can be used without the others.
- **Authoring without code.** Items, weapons, loot tables, interactions, AI
  archetypes, patrol routes, music states, and camera rigs are data assets
  built in the editor, backed by a custom dev tool and gameplay debugger
  categories.

## Design principles

1. **Composition over inheritance.** An item is a data asset holding a list of
   *fragments*, and each fragment adds one capability (stackable, equippable,
   usable, firearm). You never subclass an item; you add a fragment.
2. **Gameplay tags instead of enums.** Slots, weapon types, combat states, and
   interaction states are all `FGameplayTag`s, so adding one is a data change
   rather than a recompile.
3. **Server authority.** All meaningful mutations run on the server and
   replicate down. Clients request; the server decides.
4. **Data-driven content.** Designers author content as assets. Programmers
   extend the systems that consume them.
5. **Resolve, don't store.** Components discover their collaborators at runtime
   by walking the owner chain, so your class hierarchy doesn't have to match the
   framework's.

## Architecture

Eleven modules (ten runtime, one editor-only), each owning a clear slice and
depending only on the ones beneath it, with no circular dependencies.
`MirageFoundation` sits at the bottom with the shared vocabulary (stats,
damage, factions, spatial queries, messaging); the gameplay modules build on
it; `MirageIntegration` bridges them all at the top.

| Module | Responsibility |
| --- | --- |
| **MirageFoundation** | Shared base layer: stats & modifiers, damage pipeline, factions, target lock/registry, lag compensation, spatial hash grid, gameplay message bus. |
| **MirageIntegration** | Integration layer: base characters/PlayerState/GameMode, cross-system bridge components (stat↔GAS, AI↔combat, audio feedback), indicator UI, component injection, world-item actor. |
| **MirageInventory** | Inventory & equipment components, item data + fragments, item & context-menu subsystems, related UI. |
| **MirageCombat** | GAS combat component, weapon-state & firearm components, weapon actors, combo chains, hit-stop, action & weapon-pattern registries. |
| **MirageHitbox** | Animation-driven hit detection, reusable shapes, projectiles/beams/damage volumes, hit cue. |
| **MirageContainer** | Container component & actor, container/loot assets, seeded loot-generator subsystem. |
| **MirageInteraction** | Interactor/interactable components, server-side spatial-grid detection, interaction behavior fragments. |
| **MirageAI** | StateTree tasks/evaluators/conditions, perception & suspicion, squads, patrols, spawning & population, AI director, significance & dormancy. |
| **MirageAudio** | Music & ambient director subsystems, audio zones (volumes & splines), music states, ambient palettes, gameplay SFX. |
| **MirageCamera** | Camera director & mode component on Gameplay Cameras, rig-set assets, per-player camera settings. |
| **MirageGameFrameworkEditor** | 20+ asset types & factories, in-editor dev tool with live panels, component visualizers for patrol routes & interaction points. |

## Systems

| System | Highlights |
| --- | --- |
| **Stats & damage** | Replicated stat registry, data-driven stat definitions & modifiers, damage pipeline with hit-zone multipliers, factions, GAS bridge. |
| **Inventory** | Slot-based, weight, stacking, runtime slot expansion, drag-drop UI, data-driven context menu. |
| **Equipment** | Tag-based slots, GAS effects & granted abilities on equip, async weapon-actor attachment, equipped-vs-drawn states. |
| **Combat** | GAS abilities, input-buffered combo chains, weapon draw/sheath, firearms (ADS, reload, recoil, spread), hit-stop, notify-driven windows. |
| **Hitbox** | Per-frame & path-sweep detection from anim notifies, shape presets, projectiles, beams, damage volumes, lag-compensated rewind. |
| **Container** | Access policies (public/personal/shared), slot filters, specialized slots, seeded loot tables with weighted pools. |
| **Interaction** | Server-side spatial-grid detection, multi-point interactables, behavior fragments (instant/hold/multi-press/cooldown). |
| **AI** | StateTree behaviors, EQS tactical positioning, perception/suspicion/awareness, squads & shared stimuli, patrol routes, AI director pacing encounters. |
| **Audio** | Music director with tag-driven states, ambient director with palettes, audio zones, encounter music reacting to combat. |
| **Camera** | Gameplay Cameras rig sets, per-context camera modes, target lock framing, player camera preferences. |

## Networking

Server-authoritative: clients request, the server decides and mutates, and
changes flow back through `FFastArraySerializer` lists that fire add/change
callbacks. Nothing polls, so the UI is event-driven.

A lag-compensation subsystem keeps a rewind history of hitboxes, so ranged
validation happens against what the shooting client actually saw. GAS handles
(applied-effect, granted-ability, anim-layer tag) stay server-only and are
re-derived in `BeginPlay`, so reconnects and respawns recover the right state
with no extra RPC. The structures are also Iris-compatible.

## Roadmap

Next modules, pushing MGF toward a full RPG foundation:

- **Quest system.** Objectives, state tracking, and branching, same data-driven
  tag-based approach as the rest.
- **Dynamic movement & locomotion.** Responsive traversal combat can build on.
